# create subcommand — Plan Creation Flow

Gather ideas and requirements, then generate an executable Plans.md.

## Step 0: Check Conversation Context

If requirements can be extracted from the prior conversation, confirm:

> Choose how to create the plan:
> 1. From the prior conversation — base the plan on brainstormed content
> 2. From scratch — start with intake questions

For "from the prior conversation": extract requirements, ideas, and decisions, then confirm with the user.
After confirmation, skip to Step 3 (Technical Research).

## Step 1: Ask What to Build

If no user input is given, ask:

> What would you like to build?
>
> Examples: reservation management system / blog site / task management app / API server
>
> A rough idea is fine!

## Step 2: Increase Resolution (max 3 questions)

> Please tell me a bit more:
>
> 1. Who will use it? (Just you? A team? Public?)
> 2. Are there any services you'd like to reference?
> 3. How far do you want to take it? (MVP? Full features?)

## Step 3: Planning Quality Check

Details: `references/planning-quality.md`

Do not convert user-provided information directly into Plans.md.
When external products, competitors, specs, improvement proposals, or comparison materials are included, verify with the latest info, existing specs, memory, and TeamAgent / sub-agent multi-perspective review, and convert only the elements that should be incorporated into the task contract.
Any planning that is not a one-off or trivial task is treated as requiring TeamAgent or sub-agent involvement.

Minimum required checks:

- Latest info: prioritize WebSearch / official docs / primary sources; verify important points with multiple sources
- Existing specs: check Plans.md, README, docs, CLAUDE.md, related skills, and tests
- Memory: when harness-mem / harness-recall / `.claude/agent-memory/` / `.claude/state/` are available, check project-scoped to avoid reinventing the wheel
- Discussion: evaluate adoption value and risks from Product / Architecture / Security / QA / Skeptic perspectives
- Quality foundation: for plans with source code changes, check `formatter_baseline`; if lint / formatter is not configured, place a setup task first
- Implementation plan verification: check product fit, security fit, works in practice; reflect test / smoke / CI / review / release gates in DoDs
- Scoring: evaluate Product Fit, Evidence Strength, User Value, Implementation Feasibility, Regression Safety, Strategic Leverage, Security Safety, Works In Practice on a 5-point scale

Do not read the `harness-mem` DB directly. When search or documented memory surfaces are unavailable, explicitly state "memory not checked."
When the Task tool is unavailable, explicitly state `sub-agent not used` and evaluate the same perspectives separately.
`team_validation_mode` must be one of `not_required_lightweight` / `native` / `subagent` / `manual-pass` / `unavailable`.
For non-trivial planning, use `native` / `subagent` / `manual-pass`; do not leave `unavailable` and mark as Required.
Product / Architecture / Security / QA / Skeptic are perspective names, not agent_type names.
The Security gate does not require actually reading `.env` or secrets.

For small typo, format, README/CHANGELOG, or marker-only changes, this step may be kept light.

## Step 4: Technical Research (WebSearch)

Claude Code researches and proposes without asking the user.

```
WebSearch:
- "{{project type}} tech stack 2025"
- "{{similar service}} architecture"
```

## Step 4.4: spec.md / Plans.md Dual-Contract Check

Plans.md is the task contract that fixes "what must be done."
Root `spec.md` is the product contract that fixes "what is correct."
Do not mix these two. `/harness-plan create` is not just a Plans.md generation command — treat it as a surface that returns the co-required planning output of the spec.md product contract and Plans.md task contract.
Precedence remains `spec.md > sub-spec > Plans.md`.

`/harness-plan create` output must always be this 2-item set:

1. `Spec delta` or `Spec skip reason`
2. `Plans.md` task generation

Read root `spec.md` every time. When implementation judgment may drift, update root `spec.md` before creating Plans.md.
Do not require the user to write the spec from scratch. `Spec delta` / `Spec skip reason` are generated by Harness; the consumer only approves or revises them.
The agent drafts the minimal delta from existing specs, repo evidence, memory, tests, and input requirements; only present options when judgment is truly split.

### Conditions for creating/updating the spec authoritative source

- User-visible behavior is added or changed
- Determines APIs, data models, permissions, billing, external integrations, or tenant boundaries
- Multiple implementation approaches exist and the choice changes product behavior
- Spec ambiguity causing implementation drift is visible in past or current conversation
- Plans.md has tasks but the correct conditions at the project level are not documented

### Conditions for skipping

- typo / format / lint only
- dependency bump only
- README / CHANGELOG only
- docs-only / mechanical task
- Narrow refactor with no behavior change
- Existing specs and tests already make the correct answer clear

Even when skipping, do not omit `Spec skip reason`.
Even for docs-only / mechanical tasks, leave the skip reason in the task context / sprint contract.

### Save location

First priority is root `spec.md`.
Only when the consumer repo has no root `spec.md`, fall back to updating an existing project-level spec.
If neither root `spec.md` nor an existing project spec exists, create:

```text
docs/spec/00-project-spec.md
```

The initial spec can be short. At minimum, include Purpose, Users And Workflows, Core Rules, Data And Contracts, Non-Goals, Open Decisions, and Links.

Details: `docs/plans/spec-ssot.md`

## Step 4.6: lint / formatter Baseline Check

For plans with source code changes, check the lint / formatter baseline before creating implementation tasks.
This is not a "cleanup operation" — it is a gate to establish the foundation needed to Yes/No evaluate quality after implementation.

What to check:

- JavaScript / TypeScript: `lint` / `format` scripts in `package.json`, ESLint / Prettier / Biome / Oxlint / dprint config or dependency
- Python: Ruff / Black / isort / mypy config in `pyproject.toml`
- Go: `gofmt` / `go test` / `go vet` / equivalent lint CI command
- Rust: `cargo fmt` / `cargo clippy` / `cargo test`
- Existing CI: quality commands in `.github/workflows`, `scripts/ci/*`, `Makefile`, etc.

Leave `formatter_baseline` in the output:

```text
formatter_baseline: configured | missing | not_applicable | unknown
formatter_baseline_evidence: [file / command seen]
formatter_baseline_action: none | add_setup_task | skip_with_reason | spike
```

When not configured and source code changes are included, add a setup task to Plans.md before the implementation tasks.
The setup task DoD is: "config / script / validation command are in place, and bulk reformatting across the board is explicitly out of scope."
Do not install packages during planning. Installation is performed by harness-work as a setup task.

Conditions for skipping:

- docs-only / markdown-only / changelog-only
- An existing lint / formatter / CI command already covers the languages touched in this change
- Cannot be introduced due to consumer repo constraints. However, leave `formatter_baseline_action: spike` or a skip reason

## Step 5: Feature List Extraction

Extract a concrete feature list from the requirements.

Example: reservation management system
- User registration / login
- Reservation calendar display
- Create / edit / cancel reservations
- Admin dashboard
- Email notifications
- Payment functionality

## Step 5.5: optional brief generation

Attach a brief only when necessary. A brief is a short supplementary document that fixes the prerequisites for implementation and does not replace Plans.md.

- Tasks including UI get a `design brief`
- Tasks including APIs get a `contract brief`
- When UI and API are mixed, separate the briefs

### design brief

For UI tasks, the brief must include at minimum:

- What is to be achieved
- Who will use it
- Important screen states
- Visual and interaction constraints
- Completion conditions

### contract brief

For API tasks, the brief must include at minimum:

- What is received / returned
- Input validation conditions
- Behavior on failure
- External dependencies
- Completion conditions

## Step 6: Priority Matrix (2-axis evaluation)

Evaluate each feature on two axes: **Impact × Risk (uncertainty)**:

- **Impact**: User value × target user count (high/low)
- **Risk**: Technical unknowns × external dependencies (high/low)

| Impact \ Risk | Low Risk | High Risk |
|---------------|----------|-----------|
| **High Impact** | ★ **Required** — Highest priority (reliably delivers value) | ▲ **Required + [needs-spike]** — Early verification needed |
| **Low Impact** | ○ **Recommended** — Address when capacity allows | ✕ **Optional** — Defer or reduce scope |

### `[needs-spike]` marker

Automatically attach `[needs-spike]` to High Impact × High Risk tasks.
Tasks with `[needs-spike]` automatically get a **spike (technical verification) task** placed before them:

```markdown
| N.X-spike | [spike] Technical verification for {{task name}} | Verification result report created | - | cc:TODO |
| N.X       | {{task name}} [needs-spike] | {{DoD}} | N.X-spike | cc:TODO |
```

The completion condition for a spike task is "leave a verification result report (feasible / not feasible / requires design change)."

## Step 6.5: TDD Skip Decision (enabled by default)

TDD is enabled by default. Only attach the `[skip:tdd]` marker to tasks that fall into any of the following:

| Skip condition | Reason |
|----------------|--------|
| Documentation / comments only | No impact on executable code |
| Config files only (JSON, YAML, .env) | No logic to test |
| One-line or simpler fix (typo) | Test cost exceeds benefit |
| Style / format changes only | No impact on behavior |
| Dependency update only | No implementation logic change |
| README / CHANGELOG update | Documentation only |
| Refactoring (no behavior change) | Already covered by existing tests |

Tasks that do not fall into the above have TDD automatically applied (test-first recommended).

## Step 6.7: Plans.md v3 Format Spec

Plans.md v3 includes the following format extensions:

### Phase header Purpose line (optional)

Each Phase header can include a one-line Purpose. Omit if no input is provided:

```markdown
### Phase N.X: [Phase Name] [Px]

Purpose: [What problem this phase solves, in one line]
```

- **Default**: do not prompt for input (omit if blank)
- **Effect when present**: shown during breezing Phase 0 scope confirmation
- **Generation rule**: only auto-include when the user explicitly states the phase purpose

### Artifact notation (Status column)

Attach commit hash to Status when a task completes:

```markdown
| Task | Description | DoD | Depends | Status |
|------|-------------|-----|---------|--------|
| 1.1  | ... | ... | - | cc:done [a1b2c3d] |
| 1.2  | ... | ... | 1.1 | cc:TODO |
```

- **Format**: `cc:done [7-char hash]`
- **When attached**: automatically by `harness-work` Solo Step 7
- **Backward compatibility**: `cc:done` without hash remains valid

### Affected file list

Files related to v3 format:

| File | Impact |
|------|--------|
| `skills/harness-plan/references/create.md` | Add Purpose line to Step 6 template |
| `skills/harness-plan/references/sync.md` | Recognize `cc:done [hash]` format in discrepancy detection |
| `skills/harness-work/SKILL.md` | Attach hash in Solo Step 7, re-ticket on failure |
| `skills/harness-sync/SKILL.md` | Save snapshot with --snapshot |
| `skills/breezing/SKILL.md` | Display progress in Progress Feed |

## Step 7: Generate Plans.md

Output `Spec delta` or `Spec skip reason` first, then auto-generate quality markers + DoD + Depends and create Plans.md.

### Spec result output

`Spec delta` / `Spec skip reason` are generated by Harness; the consumer only approves or revises them.

```markdown
Spec delta:
- path: spec.md
- change: [product rule to add/change]
- why: [reason this is needed as a prerequisite for the task contract]

Plans.md:
| Task | Description | DoD | Depends | Status |
|------|-------------|-----|---------|--------|
```

```markdown
Spec skip reason:
- path checked: spec.md
- reason: [docs-only / mechanical task / correct answer already fixed by existing spec and tests]
- preserve in: task context or sprint contract

Plans.md:
| Task | Description | DoD | Depends | Status |
|------|-------------|-----|---------|--------|
```

### Quality marker assignment logic
```
Analyze task content
    ↓
├── "auth" "login" "API" → [feature:security]
├── "component" "UI" "screen" → [feature:a11y]
├── "fix" "bug" → [bugfix:reproduce-first]
├── "docs" "comment" "README" "CHANGELOG" → [skip:tdd]
├── "config" "json" "yaml" "env" → [skip:tdd]
├── "style" "format" "lint" → [skip:tdd]
├── "refactor" (no behavior change) → [skip:tdd]
├── "payment" "billing" → [feature:security]
└── other → no marker (TDD enabled by default)
```

### DoD auto-inference logic

Infer DoD from keywords in the task "description" and auto-fill:

| Task description keyword | DoD inference |
|--------------------------|---------------|
| "create" "new" "add" | File exists and has the expected structure |
| "test" | Tests pass (`npm test` / `pytest`, etc.) |
| "fix" "bug" | Problem no longer reproduces |
| "UI" "screen" "component" | Display confirmed (screenshot or browser) |
| "API" "endpoint" | Response confirmed via curl/httpie |
| "config" "configuration" | Config values are reflected |
| "documentation" "docs" | File exists, no broken links |
| "migration" "DB" | Migration is executable |
| "refactoring" | All existing tests pass + 0 lint errors |

Inferred results are default values only. When the user specifies concrete acceptance conditions, those take priority.

### Depends auto-inference logic

Infer dependencies between tasks within a phase using the following rules:

1. **DB / schema tasks** → depended on by other implementation tasks (predecessor)
2. **UI tasks** → depend on API / logic tasks (successor)
3. **Test / verification tasks** → depend on implementation tasks (last)
4. **Config / environment tasks** → depended on by other tasks (predecessor)
5. **Tasks with no clear dependency** → `-` (can run in parallel)

When inference is uncertain, use `-` and ask the user to confirm.

**Generation template**:

```markdown
# [Project Name] Plans.md

Created: YYYY-MM-DD

---

## Phase 1: [Phase Name]

Purpose: [Phase purpose (optional)]

| Task | Description | DoD | Depends | Status |
|------|-------------|-----|---------|--------|
| 1.1  | [Task description] [feature:security] | [Verifiable completion condition] | - | cc:TODO |
| 1.2  | [Task description] | [Verifiable completion condition] | 1.1 | cc:TODO |
```

**Purpose line**:
- Auto-include only when the user explicitly states the phase purpose
- If no input, omit the Purpose line entirely (do not leave a blank line)
- Keep it to one line (multiple lines forbidden)

**DoD (Definition of Done) notation**:
- Write in one verifiable line (e.g., "Tests pass", "Migration executable", "0 lint errors")
- "Looks good" or "works properly" are forbidden. It must be Yes/No evaluable

**Depends notation**:
- No dependency: `-`
- Single dependency: task number (e.g., `1.1`)
- Multiple dependencies: comma-separated (e.g., `1.1, 1.2`)
- Phase dependency: phase number (e.g., `Phase 1`)

### Team mode output

Only when the user explicitly specifies team mode, also provide a dry-run issue bridge alongside Plans.md.

- Only one tracking issue
- List per-task sub-issue payloads
- Keep Plans.md as the authoritative source
- Provide the dry-run of `scripts/plans-issue-bridge.sh --team-mode` in a usable form

## Step 8: Always provide the session launch command and first input

Immediately after presenting Plans.md, provide the **new session launch command** and
the **first input to enter immediately after launch** as a pair so the user does not hesitate on the next step.

### Rules for presenting

1. Write at least one concrete launch command + first input combination
2. When possible, narrow down to "top 1 combination + 1 alternative"
3. Add one line explaining why that combination is recommended
4. For long-running tasks, recommend `bash scripts/claude-longrun.sh` first

### Recommended mappings

| Situation | Launch command | First input |
|-----------|---------------|-------------|
| Starting from the first 1 task | `claude` | Single task execution like `/harness-work 1.1` |
| Progressing multiple tasks together | `claude` | `/breezing all` |
| Running everything sequentially | `claude` | `/harness-work all` |
| Long-running / re-entry expected | `bash scripts/claude-longrun.sh` | `/harness-loop all` |

### Output example

```text
Next step:
- New session launch command: claude
- First input after launch: /breezing all
- Best suited for: The current Plans.md is structured to progress multiple tasks together, so team execution is the most natural approach
```

```text
Next step:
- New session launch command: bash scripts/claude-longrun.sh
- First input after launch: /harness-loop all
- Best suited for: A long-running task where waits over 5 minutes or resumptions are likely to occur
```

## Step 9: Next Action Guidance

> Plans.md complete!
>
> Next steps:
> - Start implementation with `harness-work`
> - Or say "start from Phase 1"
> - Add features with `harness-plan add [feature name]`
> - Defer features with `harness-plan update [task] blocked`

## CI mode (--ci)

No intake questions. Use existing Plans.md as-is and only decompose tasks.

1. Load Plans.md
2. List cc:TODO tasks in priority order
3. Attach `[P]` mark to tasks that can run in parallel
4. Propose the next task to execute

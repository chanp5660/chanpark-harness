# Harness Plan

The unified planning skill for Harness.
Consolidates the following three legacy skills:

- `planning` (plan-with-agent) — translating ideas into Plans.md
- `plans-management` — task state management and marker updates
- `sync-status` — verifying sync between Plans.md and implementation

## Quick Reference

| User Input | Subcommand | Action |
|------------|------------|--------|
| "create a plan" / `/harness-plan create` | `create` | Spec delta / skip reason → generate Plans.md tasks |
| "add a task" / `/harness-plan add` | `add` | Add a new task to Plans.md |
| "mark complete" / `/harness-plan update` | `update` | Change task marker to cc:done |
| "where am I?" / `/harness-plan sync` | `sync` | Reconcile implementation with Plans.md |
| `/harness-sync` | `sync` | Check progress (equivalent to standalone sync surface) |
| `/harness-plan create` | `create` | Create spec.md / Plans.md dual-contract plan |
| `/harness-plan list` | `list` | List named Plans from `plans/manifest.json` |
| `/harness-plan switch <name>` | `switch` | Save active plan to `.claude/state/active-plan.json` |

## Literal companion commands (CC 2.1.108+)

- `/recap`: Re-fetch a summary after returning from a long absence, then enter `sync`
- `/undo`: Alias for `/rewind`. Use directly when you want to immediately revert the last plan update

## Subcommand Details

### Standard Planning Quality Contract

See [references/planning-quality.md](${CLAUDE_SKILL_DIR}/references/planning-quality.md)

`harness-plan` is the planning surface that produces co-required planning output for the spec.md product contract and Plans.md task contract.
Precedence remains `spec.md > sub-spec > Plans.md`.
Plans.md is the task ledger; root `spec.md` is the product contract — this hierarchy must not be collapsed.
Do not convert user-provided information directly into Plans.md without evaluation.
For plan creation or significant task additions, verify the latest information, existing specs, memory, and multi-perspective discussion via TeamAgent / sub-agents,
and convert only the elements that should be incorporated into this product into the task contract.
`/harness-plan create` returns a `Spec delta` or `Spec skip reason` together with `Plans.md` task generation.
Output must always include either `Spec delta` or `Spec skip reason`.
`Spec delta` / `Spec skip reason` are generated by Harness; the consumer only approves or revises them.

**Non-trivial planning gate**:

Any planning that is not a one-off or trivial task is treated as requiring TeamAgent or sub-agent involvement.
Non-trivial here means requests that affect multiple tasks / multiple files / multiple sessions / product behavior / APIs / data models / permissions / billing / external integrations / distribution surfaces / security.
When the Task tool is available, run independent perspectives: Product / Architecture / Security / QA / Skeptic.
When unavailable, explicitly state `sub-agent not used` and evaluate the same perspectives separately on your own.

Non-trivial planning output must include the following verifications:

- `team_validation_mode`: `not_required_lightweight` / `native` / `subagent` / `manual-pass` / `unavailable`
- Consistency across `spec.md` / sub-spec / `Plans.md`
- Reinvention-prevention check via harness-mem / harness-recall / repo memory
- Confirm the plan does not deviate from the product purpose
- Check for security, permission, secret, and supply-chain issues
- Confirm whether a lint / formatter baseline exists. For plans with source code changes, if none is set, place a setup task before any implementation tasks
- Confirm the plan is actually executable — i.e., that test / smoke / CI / review / release gates are reflected in task DoDs

Lightweight tasks may use `team_validation_mode: not_required_lightweight`.
Non-trivial planning must use `native` / `subagent` / `manual-pass`.
Do not leave `unavailable` and mark something as Required.
Product / Architecture / Security / QA / Skeptic are verification perspectives, not agent_type names.
Assign them as perspectives to available TeamAgent / Task sub-agents; do not demand arbitrary agent spawning.
The Security gate does not require actually reading secrets.
If reading `.env` or secrets would be needed, stop it as a Risk Gate and verify using permitted existing guards / evidence.

**When to apply**:

- Creating a new plan with `create`
- Adding tasks via `add` that affect product behavior / APIs / permissions / billing / external integrations / distribution surfaces
- User provides external products, competitors, specs, improvement proposals, or comparison materials
- There is a risk of conflict with existing specs or past decisions

**When to treat lightly**:

- `update` for marker-only changes
- `sync` for status reconciliation only
- typo, format, README/CHANGELOG only
- Narrow changes where existing specs and tests already fix the correct answer

**Quality flow**:
1. Decompose input information; explicitly identify what is being evaluated, the scoring axes, and uncertain facts
2. Retrieve latest information. Prioritize official documentation / WebSearch / primary sources for external facts; cross-check important points with multiple sources
3. Verify existing specs, root `spec.md`, Plans.md, README, docs, CLAUDE.md, and related skills
4. Check available memory surfaces project-scoped: harness-mem / harness-recall / `.claude/agent-memory/` / `.claude/state/`
5. For non-trivial planning, use TeamAgent / Task sub-agents to independently review from Product / Architecture / Security / QA / Skeptic perspectives
6. For plans with source code changes, check the lint / formatter baseline; if not configured, place a setup task before implementation
7. Produce a neutral scoring review and classify as Required / Recommended / Optional / Reject
8. Report in `$easy` format: proposed content, rationale, expected outcome
9. Incorporate only the accepted proposals into root `spec.md` / Plans.md / test tasks

### create — Plan Creation

See [references/create.md](${CLAUDE_SKILL_DIR}/references/create.md)

Gather ideas and requirements, then generate an executable Plans.md.

**Flow**:
1. Check conversation context (extract from prior discussion or start fresh intake)
2. Ask what to build (max 3 questions)
3. **Planning quality check** (latest info, existing specs, memory, TeamAgent / sub-agent multi-perspective review, scoring)
4. Technical research (WebSearch)
5. Extract feature list
6. **spec.md / Plans.md dual-contract check** (Spec delta or Spec skip reason + Plans.md tasks)
7. Priority matrix (Required / Recommended / Optional / Reject)
8. TDD adoption decision (test design)
9. Generate Plans.md (with `cc:TODO` markers)
10. Guide next actions

### spec.md / Plans.md dual-contract check (default)

Plans.md is treated as the task contract for "what must be done"; root `spec.md` is the product contract for "what is correct."
Co-required planning output means both outputs are mandatory; precedence remains `spec.md > sub-spec > Plans.md`.
When there is a risk of implementation drift, update root `spec.md` before generating Plans.md.
`create` and product-impacting `add` must read root `spec.md` every time.

Preferred save locations:

1. root `spec.md`
2. Only when consumer repo has no root `spec.md`: existing project spec / architecture / product compass
3. Only when consumer repo has no root `spec.md`: `docs/spec/00-project-spec.md`
4. For repos with existing conventions, follow the spec path that matches those conventions

Conditions requiring creation/update:

- Tasks that determine user-visible behavior, APIs, data models, permissions, billing, or external integrations
- Tasks with multiple implementation approaches where the choice changes product behavior
- Tasks where there are signs of "spec ambiguity causing implementation drift" in past or current conversation
- Tasks that have work items in Plans.md but no stable document defining the correct answer at the project level

Conditions where it is not required:

- typo, format, dependency bump, README/CHANGELOG only
- Narrow refactor with no behavior change
- Narrow fixes where existing specs and tests already fully fix the correct answer

Output contract:

- `Spec delta`: when updating the product contract, write the target spec path and the change
- `Spec skip reason`: when not updating the product contract, write the reason
- `Spec delta` / `Spec skip reason` are generated by Harness; the consumer only approves or revises them
- Even for docs-only / mechanical tasks, leave `Spec skip reason` in the task context / sprint contract
- Do not assert absent for missing search results, unavailable memory, or unread files. `not_observed != absent`
- Do not require the user to write the spec from scratch. The agent drafts the minimal delta from existing specs and input; only present decision branches when truly ambiguous

Reference:

- `docs/plans/spec-ssot.md`

### Session launch guidance after create (required)

After `create` completes, do not end with explanation alone — provide both the **new session launch command** and
the **first input to enter immediately after launch** as a pair.

Priority order:

1. Only one incomplete task, or it is natural to start just the first one
   - Launch command: `claude`
   - First input: `/harness-work <task number>`
2. Multiple tasks with loose dependencies where progressing together is natural
   - Launch command: `claude`
   - First input: `/breezing all`
   - Alternative: `/harness-work all`
3. Long-running execution or re-entry is expected
   - Launch command: `ENABLE_PROMPT_CACHING_1H=1 claude`
   - First input: `/harness-loop all`
   - Alternative: `/breezing all`

Include at minimum the following 3 lines:

- `New session launch command:`
- `First input after launch:`
- `Best suited for:`

Example:

```text
New session launch command: claude
First input after launch: /breezing all
Best suited for: The current Plans.md has multiple Phase 1 tasks, so team execution is the most natural approach
```

When recommending long-running mode, also include the Claude Code session launch command:

```text
New session launch command: ENABLE_PROMPT_CACHING_1H=1 claude
First input after launch: /harness-loop all
Best suited for: Long-running tasks that involve waits over 5 minutes or cross-session resume
```

Note:

- `scripts/claude-longrun.sh` is a development helper script for this repository and is not distributed to consumer environments after plugin install
- Therefore, for consumer-facing guidance, always prefer the single-line command `ENABLE_PROMPT_CACHING_1H=1 claude`
- During repository development only, the equivalent wrapper `bash scripts/claude-longrun.sh` may be used on a local checkout

**CI mode** (`--ci`):
No intake questions. Use existing Plans.md as-is and only decompose tasks.

### add — Add Tasks

Add new tasks to Plans.md.
For product-impacting additions, output `Spec delta` or `Spec skip reason` as well, following the "spec.md / Plans.md dual-contract check" above.

```
/harness-plan add <task name>: <detailed description> [--phase <phase number>]
```

Tasks are added with the `cc:TODO` marker.

### update — Marker Update

Change the status marker of a task.

```
/harness-plan update [task name|task number] [WIP|done|blocked]
```

Marker mapping:

| Command | Marker |
|---------|--------|
| `WIP` | `cc:WIP` |
| `done` | `cc:done` |
| `blocked` | `blocked` |
| `TODO` | `cc:TODO` |

### sync — Progress Sync

Reconcile implementation status with Plans.md, detect and update discrepancies.

See [references/sync.md](${CLAUDE_SKILL_DIR}/references/sync.md)

**Flow**:
1. Retrieve current Plans.md state
2. Detect Plans.md format (v1: 3 columns / v2: 5 columns)
3. Retrieve implementation status from git status / git log
4. Check agent trace (`.claude/state/agent-trace.jsonl`)
5. Detect discrepancies between Plans.md and implementation
6. Propose automatic corrections for outdated markers
7. Present next actions

**Retrospective** (default ON):
If one or more `cc:done` tasks exist, automatically run a retrospective.
Analyze estimation accuracy, blocker patterns, and scope changes, then record the learnings.
Can be explicitly skipped with `sync --no-retro`.

### team mode / issue bridge

Keep Plans.md as the authoritative source; GitHub Issue integration is opt-in and used only in team mode.

- Do not use the bridge for solo development
- Team mode creates one tracking issue and generates per-task sub-issue payloads as a dry-run
- `scripts/plans-issue-bridge.sh` never actually updates GitHub — it always returns a dry-run payload
- This bridge does not make changes to Plans.md

Reference:

- `docs/plans/team-mode.md`

### named Plans

When using multiple Plans.md files, use `plans/manifest.json` as the authoritative source and select by name.

```bash
scripts/plan-registry.sh list
scripts/plan-registry.sh switch roadmap
scripts/plans-issue-bridge.sh --plan roadmap --format markdown
node scripts/generate-sprint-contract.js --plan roadmap 9.1.1
```

Operating rules:

- Use only one named plan per run
- For long-running / CI / issue bridge, pass `--plan <name>` explicitly rather than relying on the active pointer
- Manifest paths must be relative to the project root only. Absolute paths, `..`, and out-of-repo symlinks are rejected

## Plans.md Format Conventions

### Format

```markdown
# [Project Name] Plans.md

Created: YYYY-MM-DD

---

## Phase N: Phase Name

| Task | Description | DoD | Depends | Status |
|------|-------------|-----|---------|--------|
| N.1  | Description | Tests pass | - | cc:TODO |
| N.2  | Description | 0 lint errors | N.1 | cc:WIP |
| N.3  | Description | Migration executable | N.1, N.2 | cc:done |
```

**DoD (Definition of Done)**: Write the completion condition in one verifiable line. "Looks good" or "works properly" are forbidden. It must be Yes/No evaluable.

**Depends**: Dependencies between tasks. `-` (no dependency), task number (`N.1`), comma-separated (`N.1, N.2`), or phase dependency (`Phase N`).

### TDD tags

Plans.md tasks can include TDD decision tags in the description or DoD.

| Tag | Meaning | `tdd_required` inference |
|-----|---------|--------------------------|
| `[tdd:required]` | This task requires writing a failing test first | `true` |
| `[tdd:skip:<reason>]` | This task skips TDD with a stated reason | `false`, `skip_tdd_reason=<reason>` |

`<reason>` must not be empty.
Examples: `[tdd:skip:docs-only]`, `[tdd:skip:no-test-framework-detected]`.

When no tag is present, `tdd_required` is inferred in the following order:

1. Plans.md tag: `[tdd:required]` / `[tdd:skip:<reason>]`
2. files: required if source implementation is included under `src/`, `app/`, `cmd/`, `lib/`, `pkg/`, `internal/`, `go/`, etc.
3. TDD inference: not required if docs-only or no test framework; attach a skip reason

### optional briefs / manifest

`harness-plan create` attaches a brief only when necessary.

- A project spec SSOT is a document that fixes the correct conditions for the entire project; create it only when needed
- Tasks including UI get a `design brief`
- Tasks including APIs get a `contract brief`
- A brief is a short supplementary document that fixes "what to build" and does not replace Plans.md or the spec SSOT
- The skill frontmatter list can be converted to machine-readable JSON with `scripts/generate-skill-manifest.sh`

Reference:

- `docs/plans/briefs-manifest.md`
- `docs/plans/spec-ssot.md`

|--------|---------|
